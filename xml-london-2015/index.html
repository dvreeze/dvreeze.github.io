What is yaidom?

  • An XML query and transformation API
  • Leverages Scala and the Scala Collections API
  • The XML query API is small, precise and clear
  • It is offered by multiple element implementations
  • Including your own custom ones (easy to add)
  • Including type-safe ones for an XML dialect (e.g. XBRL)
  • Very precise XML namespace support
  • Full control over XML parsing and serialization
  • In short, a useful XML library when using Scala to write complex (XML) validations

  • Yaidom is shown using the XBRL example below
  • XBRL is a (financial) reporting standard

Introducing Scala and Scala Collections

Consider the following trivial (non-XML) instance model: 


trait Fact { def ename: EName }
							

final class ItemFact(
  val ename: EName,
  val contextRef: String,
  val unitRefOption: Option[String],
  val factValue: String) extends Fact
							

final class TupleFact(
  val ename: EName,
  val childFacts: immutable.IndexedSeq[Fact]) extends Fact {

  def childItemFacts: immutable.IndexedSeq[ItemFact] =
    childFacts collect { case item: ItemFact => item }

  def childTupleFacts: immutable.IndexedSeq[TupleFact] =
    childFacts collect { case tuple: TupleFact => tuple }
}

The trivial (non-XML) instance model itself: 


final class XbrlInstance(
  val topLevelFacts: immutable.IndexedSeq[Fact]) {

  // Contexts, units etc. ...

  def topLevelItemFacts: immutable.IndexedSeq[ItemFact] =
    topLevelFacts collect { case item: ItemFact => item }

  def topLevelTupleFacts: immutable.IndexedSeq[TupleFact] =
    topLevelFacts collect { case tuple: TupleFact => tuple }
}


Higher-order function "collect" acts as a combination of "filter" and "map", explained below.

The example instance data in this model looks as follows: 


val instance =
  new XbrlInstance(Vector(
      new ItemFact(
        EName("cc2-i", "Equity"), "FY14d", Some("EUR"), "95000"),
      new TupleFact(
        EName("cc-t", "EntityAddressPresentation"), Vector(
          new ItemFact(
            EName("nlcd", "POBoxNumber"), "FY14d", None, "2312"),
          new ItemFact(
            EName("nlcd", "PostalCodeNL"), "FY14d", None, "2501CD"),
          new ItemFact(
            EName("nlcd", "PlaceOfResidenceNL"), "FY14d", None,
              "Den Haag"),
          new ItemFact(
            EName("nlcd", "CountryName"), "FY14d", None,
              "Nederland")))))

Showing higher-order function "filter": 


val topLevelEuroFacts =
  instance.topLevelItemFacts.filter(
    fact => fact.unitRefOption == Some("EUR"))

require(topLevelEuroFacts.size == 1)

// Using a for-comprehension

val topLevelEuroFacts2 =
  for {
    itemFact <- instance.topLevelItemFacts
    if itemFact.unitRefOption == Some("EUR")
  } yield itemFact

require(topLevelEuroFacts2 == topLevelEuroFacts)

Showing higher-order function "map": 


val address = instance.topLevelTupleFacts.head

val addressContextRefs =
  address.childItemFacts.map(fact => fact.contextRef).toSet

require(addressContextRefs == Set("FY14d"))

// Using a for-comprehension

val addressContextRefs2 =
  (for {
    childItem <- address.childItemFacts
  } yield childItem.contextRef).toSet

require(addressContextRefs2 == addressContextRefs)

Introducing yaidom

  • Let's now return to XML, and let's introduce yaidom
  • Suppose we want to perform some (instance) validations
  • Then we need to be able to query the XBRL instance
  • Let's query the instance at the low abstraction level of XML
  • Using higher-order yaidom functions returning element collections
  • E.g., using HOF filterChildElems to get contexts, units and top-level facts
  • And using HOF filterElems (from top-level facts) to get nested facts ("filter descendant elements")
  • And using HOF filterElemsOrSelf (from top-level facts) to get all facts ("filter descendant-or-self elements")

Finding facts, contexts and units (as plain XML elements), regardless of the element implementation: 


val ns = "http://www.xbrl.org/2003/instance"
val linkNs = "http://www.xbrl.org/2003/linkbase"

def hasCustomNs(e: Elem): Boolean = {
  !Set(Option(ns), Option(linkNs)).contains(
    e.resolvedName.namespaceUriOption)
}

val contexts = xbrlInstance.filterChildElems(withEName(ns, "context"))
val units = xbrlInstance.filterChildElems(withEName(ns, "unit"))
val topLevelFacts =
  xbrlInstance.filterChildElems(e => hasCustomNs(e))
val nestedFacts =
  topLevelFacts.flatMap(_.filterElems(e => hasCustomNs(e)))
val allFacts =
  topLevelFacts.flatMap(_.filterElemsOrSelf(e => hasCustomNs(e)))

Non-trivial queries combine facts with their contexts and units: just store contexts and units in Maps

Namespace validation example

  • To illustrate (low level) validations, let's check the use of "standard" namespaces
  • In particular, let's validate rule 2.1.5 of the international FRIS standard
  • The rule states that some commonly used namespaces should use their "preferred" prefixes in XBRL instances
  • For simplicity, assume that all namespace declarations are only in the root element
<xbrli:xbrl xmlns:xbrli="http://www.xbrl.org/2003/instance"

  xmlns:cc2-i="cc2i" xmlns:cc-t="cct" xmlns:cd="nlcd" xmlns:iso4217="iso4217">

  <xbrli:context id="FY14d"> 

    <xbrli:entity>

      <xbrli:identifier scheme="http://www.cc.eu/cc-id">30267975

      </xbrli:identifier>

    </xbrli:entity>

    <xbrli:period>

      <xbrli:startDate>2014-01-01</xbrli:startDate>

      <xbrli:endDate>2014-12-31</xbrli:endDate>

    </xbrli:period>

  </xbrli:context>

  <xbrli:unit id="EUR">

    <xbrli:measure>iso4217:EUR</xbrli:measure>

  <cc2-i:Equity contextRef="FY14d" unitRef="EUR"

// All namespace declarations must be in the root element

require(
  xbrlInstance.findAllElems.forall(_.scope == xbrlInstance.scope))
							

val standardScope = Scope.from(
  "xbrli" -> "http://www.xbrl.org/2003/instance",
  "xlink" -> "http://www.w3.org/1999/xlink",
  "link" -> "http://www.xbrl.org/2003/linkbase",
  "xsi" -> "http://www.w3.org/2001/XMLSchema-instance",
  "iso4217" -> "http://www.xbrl.org/2003/iso4217")

val standardPrefixes = standardScope.keySet
val standardNamespaceUris = standardScope.inverse.keySet
							

val subscope = xbrlInstance.scope.withoutDefaultNamespace filter {
  case (pref, ns) =>
    standardPrefixes.contains(pref) ||
      standardNamespaceUris.contains(ns)
}
require(subscope.subScopeOf(standardScope)) // fails on iso4217

XBRL context validation example

  • Let's now validate rule 2.4.2 of the international FRIS standard
  • The rule states that all contexts must be used
  • We also check the reverse, that all context references indeed refer to existing contexts
  • N.B. The latter check belongs to XBRL instance validation, not to FRIS validation for XBRL-valid instances


val ns = "http://www.xbrl.org/2003/instance"
val linkNs = "http://www.xbrl.org/2003/linkbase"

def hasCustomNs(e: Elem): Boolean = {
  !Set(Option(ns), Option(linkNs)).contains(
    e.resolvedName.namespaceUriOption)
}

val contexts = xbrlInstance.filterChildElems(withEName(ns, "context"))
val units = xbrlInstance.filterChildElems(withEName(ns, "unit"))
val topLevelFacts =
  xbrlInstance.filterChildElems(e => hasCustomNs(e))
val allFacts =
  topLevelFacts.flatMap(_.filterElemsOrSelf(e => hasCustomNs(e)))
							

val contextIds =
  contexts.map(_.attribute(EName("id"))).toSet

val usedContextIds =
  allFacts.flatMap(_.attributeOption(EName("contextRef"))).toSet

require(usedContextIds.subsetOf(contextIds))
require(contextIds.subsetOf(usedContextIds))

XBRL context validation example, revisited

  • Let's hint at how to implement the same rule at a higher level of abstraction
  • Yaidom makes it easy to create class hierarchies for specific XML dialects that are yaidom elements themselves
  • So these custom yaidom elements offer the same yaidom query API, plus more
  • For example, assume such a class hierarchy for XBRL instances and their components
  • Then the validation code above could be reduced to something like the code shown below

Clear and type-safe validation code, using an imaginary yaidom extension for XBRL instances: 


val contextIds = xbrlInstance.allContextsById.keySet

val usedContextIds =
  xbrlInstance.findAllItems.map(_.contextRef).toSet

require(usedContextIds.subsetOf(contextIds))
require(contextIds.subsetOf(usedContextIds))

The code looks a lot more like client code of the non-XML instance model shown earlier.

Yet here we can treat the instance like (type-safe) XML.

Takeaway points about yaidom

  • Like the standard Scala XML library, yaidom leverages Scala and its Collections API
  • Yet yaidom offers multiple element implementations behind the same query API
  • Including James Clark's minimal "resolved" element trees
  • Or custom ones, such as Saxon NodeInfo wrappers
  • Even custom XML dialect element implementations are easy to add using yaidom
  • In short, yaidom and Scala are a foundation for (complex) XML validations
  • At EBPI, yaidom is one tool through which we achieve high data quality and a quick time to market
